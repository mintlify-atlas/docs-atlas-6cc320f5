Andurel is inspired by Ruby on Rails but deliberately limits how much convention it imposes. The goal is _just enough_ structure to eliminate boilerplate without making the framework feel opinionated about things that don't matter.

## Naming conventions

### Models and tables

Model names are **PascalCase** (e.g., `Product`, `BlogPost`). Andurel automatically maps these to **snake_case plural** table names when generating queries and migrations.

| Model name | Table name |
| ---------- | ---------- |
| `Product` | `products` |
| `BlogPost` | `blog_posts` |
| `OrderItem` | `order_items` |

You can override the inferred table name when generating a resource:

```bash
andurel generate resource Product --table-name catalog_items
```

### Files and packages

- Go files follow standard Go conventions: `snake_case` filenames, `PascalCase` exported identifiers.
- Controller files are named after the resource they handle (e.g., `products_controller.go`).
- Migration files are timestamped by Goose and should have a descriptive suffix (e.g., `20240101120000_create_products_table.sql`).

## Generated code ownership

Andurel draws a clear line between code it manages and code that belongs to you.

### Code you own — edit freely

Files created by `andurel generate` are yours from the moment they're written. Andurel will never overwrite them. This includes:

- Controllers in `controllers/`
- Views in `views/` (the `.templ` source files)
- Models in `models/` (excluding `models/internal/db/`)
- Routes in `router/routes/`
- Jobs and workers in `queue/`
- Services in `services/`

### Code you must not edit

Two locations contain fully generated files that are overwritten every time you compile:

- **`models/internal/db/`** — generated by SQLC when you run `andurel query compile`. Edit the SQL in `database/queries/` instead.
- **`views/*_templ.go`** — generated by Templ when you run `andurel view generate`. Edit the `.templ` source files instead.

<Warning>
  Any manual changes to `models/internal/db/` or `*_templ.go` files will be silently overwritten the next time the generator runs.
</Warning>

## The `andurel.lock` file

`andurel.lock` is a JSON file at the root of your project that pins the versions of all framework-managed tools:

- `goose` — migration runner
- `sqlc` — query compiler
- `templ` — template compiler
- `shadowfax` — dev server
- `tailwindcli` — CSS compiler
- `mailpit`, `dblab`, `usql` — optional development utilities

When you run `andurel tool sync`, Andurel downloads any missing binaries at the exact versions recorded in the lock file. Run `andurel tool set-version <tool> <version>` to upgrade a specific tool.

**Commit `andurel.lock` to version control.** It ensures every developer and CI environment uses the same tool versions without any manual coordination.

## Environment configuration

Configuration is loaded from a `.env` file using [godotenv](https://github.com/joho/godotenv) and mapped to typed structs via [caarlos0/env](https://github.com/caarlos0/env). Copy `.env.example` to `.env` and fill in your values before running the application.

```bash
cp .env.example .env
```

The `.env` file is listed in `.gitignore` by default. Never commit it.

## Controllers and routes

Controllers follow Echo v5's handler signature. The context parameter is a pointer (`*echo.Context`):

```go
func (u Users) Index(etx *echo.Context) error {
    // ...
}
```

Routes are declared as typed variables in `router/routes/`, which lets you reference them by name when building URLs or issuing redirects — no hardcoded path strings.

## Resource generation is a one-time scaffold

Running `andurel generate resource Product` creates a complete CRUD interface: model, controller, views, and routes. This is intentionally a **one-time operation**. Andurel will not regenerate or update these files if you run the command again. Once generated, the code is yours to extend and maintain.
