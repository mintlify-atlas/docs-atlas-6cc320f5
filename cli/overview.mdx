The Andurel CLI is the primary interface for creating projects, generating code, managing databases, and running development tools. Every command is available from any directory inside an Andurel project (the CLI locates the project root via `go.mod`).

## Usage

```bash
andurel [command] [subcommand] [flags]
```

Run `andurel` with no arguments to print the banner and command list.

## Additional commands

| Command | Description |
|---------|-------------|
| `andurel app console` (alias: `a c`) | Open an interactive SQL console against the project database using `usql`. Reads credentials from `.env`. |
| `andurel app dblab` (alias: `a d`) | Open a terminal database UI (dblab) connected to the project database. |
| `andurel app mailpit` (alias: `a m`) | Start the Mailpit local email server. SMTP on port 1025, web UI on port 8025. |
| `andurel tool sync` (alias: `t`) | Download and validate all binaries listed in `andurel.lock`. See [tool](/cli/tool). |
| `andurel llm` | Print comprehensive framework documentation for AI assistants. Subcommands: `controllers`, `models`, `views`, `router`, `hypermedia`, `jobs`, `config`. |

## Global behavior

- Commands that require a project context walk up the directory tree to find `go.mod`. You do not need to be in the project root.
- Commands that call tool binaries (goose, sqlc, templ, shadowfax) expect those binaries to exist under `bin/` in the project root. Run `andurel tool sync` after creating a new project or cloning an existing one.
- Database commands read credentials from the `.env` file at the project root using the variables `DB_KIND`, `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`, and `DB_SSL_MODE`.
