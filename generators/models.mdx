---
title: "Model generator"
description: "Generate a model file with CRUD database functions and an optional test factory."
---

The model generator creates a Go model file with full CRUD database functions backed by SQLC-generated code, along with a factory file for generating test data.

## Usage

```bash
andurel generate model [name]
```

**Alias:** `andurel g m [name]`

| Flag | Description |
|------|-------------|
| `--table-name` | Override the default table name (defaults to the plural form of the model name) |
| `--skip-factory` | Skip factory file generation |

## Table name inference

By default, Andurel derives the table name from the model name by pluralizing and snake-casing it:

| Model name | Default table name |
|------------|--------------------|
| `User` | `users` |
| `Product` | `products` |
| `Article` | `articles` |
| `BlogPost` | `blog_posts` |

Use `--table-name` when your table name does not follow the default convention.

## Examples

```bash
# Generate a model for the 'users' table
andurel generate model User

# Generate a model using a custom table name
andurel generate model User --table-name=accounts

# Generate a model without a factory
andurel generate model Product --skip-factory
```

## What gets created

Running `andurel generate model User` produces:

| File | Description |
|------|-------------|
| `models/user.go` | Model with CRUD functions wrapping SQLC-generated code |
| `models/factories/user.go` | Factory for generating test data (unless `--skip-factory` is passed) |
| `database/queries/users.sql` | SQLC query definitions for the table |
| `models/internal/db/` | Generated SQLC Go code (do not edit) |

After writing the query file, the generator automatically runs `sqlc generate` to produce the type-safe database access code in `models/internal/db/`.

## Prerequisites

A migration defining the table must exist in `database/migrations/` and the table must have been applied (`andurel migrate up`) before generating a model. The generator reads your migration history to understand the table schema.

<Note>
The files under `models/internal/db/` are generated by SQLC and should not be edited manually. Your model file in `models/` wraps this generated code and is yours to extend.
</Note>

## Factory files

Factory files live in `models/factories/` and are used to create model instances with sensible default values in tests. They are generated automatically unless you pass `--skip-factory`.

If factory generation fails (for example, due to an unsupported column type), the model and SQL files are still written and a warning is printed. Factory generation failure does not abort the entire command.
