title: "Migrations"
description: "Manage database schema changes with Goose SQL migrations via the andurel migrate command."
---

Andurel uses [Goose](https://github.com/pressly/goose) for SQL migrations. Migration files live in `database/migrations/` as plain SQL files, and the `andurel migrate` command wraps the Goose binary bundled at `bin/goose`.

<Info>
  The Goose binary is managed by Andurel. If it is missing, run `andurel tool sync` to download it.
</Info>

## Commands

| Command | Description |
|---|---|
| `andurel migrate new [name]` | Create a new SQL migration file |
| `andurel migrate up` | Apply all pending migrations |
| `andurel migrate down` | Roll back the most recent migration |
| `andurel migrate status` | Show current migration status |
| `andurel migrate fix` | Re-number migrations to fix gaps |
| `andurel migrate reset` | Roll back all migrations and reapply them |
| `andurel migrate up-to [version]` | Apply migrations up to a specific version |
| `andurel migrate down-to [version]` | Roll back migrations to a specific version |

The `migrate` command also has aliases `m` and `mig`.

## Migration file format

Migration files use the Goose SQL format with `-- +goose Up` and `-- +goose Down` markers to separate the forward and rollback SQL.

```sql 20240115120000_create_products_table.sql
-- +goose Up
CREATE TABLE products (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        TEXT NOT NULL,
    description TEXT,
    price_cents INTEGER NOT NULL DEFAULT 0,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- +goose Down
DROP TABLE IF EXISTS products;
```

File names are prefixed with a timestamp version number generated by `andurel migrate new`.

## Typical migration workflow

<Steps>
  <Step title="Create a migration file">
    Run `andurel migrate new` with a descriptive name for the change.

    ```bash
    andurel migrate new create_products_table
    ```

    This creates a new file in `database/migrations/` with the Goose SQL skeleton ready to fill in.
  </Step>

  <Step title="Write the SQL">
    Open the generated file and add your SQL under the `-- +goose Up` and `-- +goose Down` markers.
  </Step>

  <Step title="Apply the migration">
    ```bash
    andurel migrate up
    ```
  </Step>

  <Step title="Verify the status">
    ```bash
    andurel migrate status
    ```

    This lists every migration file and whether it has been applied.
  </Step>
</Steps>

## Targeted migrations

Apply or roll back to a precise version by passing the version number (the timestamp prefix on the file name):

```bash
andurel migrate up-to 20240115120000
andurel migrate down-to 20240115120000
```

## Fixing migration gaps

If migration version numbers have gaps — for example after rebasing branches — run:

```bash
andurel migrate fix
```

This re-numbers the migration files sequentially so Goose applies them in the correct order.

## Resetting migrations

```bash
andurel migrate reset
```

This rolls back every applied migration and then reapplies them all from scratch. Useful during development to verify that your `-- +goose Down` SQL is correct.

<Tip>
  Always create a migration before generating a resource. The `andurel generate resource` command reads the table schema from your migrations to produce correct model code.
</Tip>
